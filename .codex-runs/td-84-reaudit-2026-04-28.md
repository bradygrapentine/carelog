## Critical (16)

### Auth / RLS bypass via supabaseAdmin without membership check

- **[apps/web/server/routers/medications.ts:201-225]** `tagEvent` / `untagEvent` / `getTagsForEvent` accept `care_event_id` + `org_id` from input and call service-role helpers with NO caller membership check — any authenticated user can write/read medication tags on any org's care events by guessing UUIDs. `untagEvent` and `getTagsForEvent` don't even gate on `org_id`. **Fix:** call `requireCoordinator(input.org_id, ctx.user.id)` (or membership gate) before all four tag procedures, and verify the target care_event/tag belongs to that org before mutating.
- **[apps/web/server/routers/notifications.ts:14-32]** `registerToken` upserts `push_tokens` with `onConflict: "token"` setting `auth_user_id` to the caller — a malicious caller can pass another user's existing device token to steal that row (re-binding to attacker's auth_user_id) and hijack push delivery. **Fix:** scope the upsert to `.eq("auth_user_id", ctx.user.id)` on update, or upsert on `(auth_user_id, token)` and reject collisions where existing token belongs to a different user.
- **[apps/web/server/routers/medications.ts:255-275]** `get` reads via RLS (good) but then calls `listEventsForMedication` / `listDocumentsForMedication` (supabaseAdmin) with NO org check — leaks medication-linked events/documents cross-org. Same issue for `getEventIdsForMedication` / `getDocumentIdsForMedication` (lines 277-289), which don't even read the medication first. **Fix:** gate by membership on the medication's org_id before calling the service-role helpers.
- **[apps/web/server/routers/careEventComments.ts:23-45 + repositories/careEventCommentsRepository.ts:137-145]** `add` resolves event's org_id via `getEventOrgId` (supabaseAdmin) then passes it straight into `insertComment` — no membership check that ctx.user belongs to that org. RLS on insert is the only gate. **Fix:** assert `membership(ctx.user.id, orgId)` before insert.
- **[apps/web/server/repositories/messagesRepository.ts:71-93]** `findOrCreateDm` is service-role and uses caller-supplied `targetUserId` + `orgId` with no check that caller is a member of `orgId` or that `targetUserId` is also a member. Lets any authenticated user create DM threads with arbitrary users in arbitrary orgs. **Fix:** assert both users have accepted memberships in orgId before insert.
- **[apps/web/server/repositories/messagesRepository.ts:96-116]** `createGroupThread` (called from `messages.createGroup` with no auth check) — identical risk: service-role insert + arbitrary `memberUserIds`, no membership verification of caller or members in `orgId`. **Fix:** gate on caller membership and filter `memberUserIds` to verified members of `orgId`.
- **[apps/web/server/routers/shiftTradeRequests.ts:67-81 + repositories/shiftTradeRequestsRepository.ts:81-132]** `respond` action="accept" calls `acceptRequest` (service role) with `requestId` + `acceptingUserId` — no check that the accepting user is the trade target or a member of the trade's org. Service role then reassigns the shift. **Fix:** in `acceptRequest`, fetch trade.target_user_id + trade.org_id and assert acceptingUserId === target_user_id (or coordinator), before update.

### Missing CSRF/rate-limit on mutating REST routes

- **[apps/web/app/api/referral/track/route.ts:5]** `POST` — missing `rateLimit()` and Zod schema (raw destructure of `body.orgId` from `request.json()` with type assertion), no origin check. Cookies auto-send → CSRF spam under any logged-in user. **Fix:** `rateLimit(request, 'referral/track')`, define a Zod schema, validate `request.headers.get('origin')` against an allowlist.
- **[apps/web/app/api/push/web-subscribe/route.ts:5]** `POST` — no rateLimit, raw destructure of `endpoint`/`keys` (only truthy checks, no Zod), no origin check. Attacker on victim's browser can register an attacker-controlled push endpoint to receive that user's PHI push notifications. **Fix:** rateLimit + Zod (validate URL host of endpoint) + origin allowlist.
- **[apps/web/app/api/push/web-subscribe/route.ts:36]** `DELETE` — no rateLimit, no Zod, no origin check. CSRF can wipe a victim's push subscription causing silent notification loss for medication reminders. **Fix:** rateLimit + Zod + origin check.
- **[apps/web/app/api/outer-circle/[shareToken]/claim/route.ts:5]** `POST` — no Zod (raw `body.name`/`email`/`note`/`slot_date` typed via `as`); intentionally public (share token) but lacks structural validation, malformed payloads reach the RPC. **Fix:** Zod schema with `.max()` bounds + `z.string().email()`.
- **[apps/web/app/api/brief/route.ts:12]** `POST` — missing `rateLimit()` entirely; auth + Zod present, no origin check, no rate limit on a route hitting supabaseAdmin and de-tokenizing identity_vault. Abusive caller can grind PHI lookups. **Fix:** `rateLimit(request, 'brief/create')` + origin allowlist.
- **[apps/web/app/api/stripe/checkout/route.ts:14]** `POST` — missing `rateLimit()`, no origin check; cookie-auth on a checkout-session-creation endpoint = CSRF can create surprise Stripe sessions billed to the victim's org. **Fix:** rateLimit + origin allowlist.
- **[apps/web/app/api/stripe/portal/route.ts:13]** `POST` — missing `rateLimit()`, no origin check; CSRF can mint billing-portal URLs and leak them via Referer if rendered in an attacker page. **Fix:** rateLimit + origin allowlist.

### Silent failure (Inngest — kills retries)

- **[apps/web/inngest/functions/documentsExtractText.ts:88-96]** `download-file` step swallows storage error — `if (error || !data) return null` then handler returns `{ ok: false, reason: "file download failed" }`. Inngest never retries; document silently never gets text-extracted. **Fix:** `throw` inside step.run when error is set so Inngest retries.
- **[apps/web/inngest/functions/educationTipRefresh.ts:14]** DB error swallowed — `if (error || !orgs) return { refreshed: 0 }` returns success on DB failure. Inngest will not retry; daily tip refresh silently fails forever. **Fix:** `throw` on `error`.

## Synthesis & next steps

**Patterns worth fixing as a batch (highest leverage):**

1. **supabaseAdmin without membership gate** (7 Critical) — top systemic risk. All 7 share the same pattern: caller-supplied `org_id`/`recipient_id`/`care_event_id` flows into a service-role helper without `assertMembership`/`requireCoordinator`. Single-PR fix: add a shared `assertCallerInOrg(orgId, userId)` helper and audit every supabaseAdmin call site.
2. **Zero origin/CSRF protection across all mutating REST routes** (1 systemic Medium + several specific) — add a shared `assertSameOrigin(request)` helper, gate POST/PUT/PATCH/DELETE in `proxy.ts` for defense-in-depth.
3. **Missing rateLimit on 5 mutating routes** — straightforward additions: `referral/track`, `push/web-subscribe` (POST + DELETE), `brief`, `stripe/checkout`, `stripe/portal`, `history/export/pdf`.
4. **Inngest .error swallowing** (~10 sites, all Medium) — pattern fix: every `supabaseAdmin.{insert,update}` inside `step.run` must destructure `{ error }` and throw. Prevents silent partial state.
5. **Migration idempotency** (~10 sites, Medium) — bulk fix: add `DROP POLICY IF EXISTS` before every `CREATE POLICY`, `IF NOT EXISTS` to every `CREATE INDEX`.
