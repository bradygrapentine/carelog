# Carelog — Security Model

## PHI Boundary

Real names, dates of birth, and contact information live only in `identity_vault`.
Everything else uses an opaque `identity_token` UUID.

| Table | What it stores | Contains real name? |
|---|---|---|
| `identity_vault` | full_name, DOB, contact_info | **Yes — PHI** |
| `care_recipients` | identity_token (UUID reference), diagnoses, allergies | No |
| `display_names` | 24h cache of full_name per recipient_id | Yes (cached copy) |
| `care_events` | jsonb payload, actor_id, recipient_id | No |
| `care_briefs` | de-tokenized snapshot at creation time | Yes (snapshot) |

**What this prevents:** Real names never appear in:
- Application logs
- Sentry error reports
- PostHog analytics events
- Inngest job payloads
- Network requests from client components

All of these systems only see UUIDs.

## Service Role Key Isolation

`SUPABASE_SERVICE_ROLE_KEY` bypasses ALL Row Level Security.

```
Allowed import locations:
  apps/web/server/supabaseAdmin.server.ts   ← definition
  apps/web/server/repositories/*            ← via supabaseAdmin import
  apps/web/app/api/*                        ← direct use in route handlers

Never allowed:
  apps/web/app/**/page.tsx
  apps/web/app/**/layout.tsx
  apps/web/components/**
  apps/mobile/**
  packages/**
  Anything prefixed NEXT_PUBLIC_
```

**Runtime guard** in `supabaseAdmin.server.ts`:
```typescript
if (typeof window !== 'undefined') {
  throw new Error('supabaseAdmin.server.ts imported on the client...')
}
```

This throws immediately if the file is accidentally bundled into the browser,
catching the mistake at runtime rather than silently leaking the key.

## Row Level Security Design

All RLS policies use **scalar boolean functions**, never set-returning functions.

```sql
-- WRONG — Postgres rejects set-returning functions in USING clauses (SQLSTATE 0A000)
CREATE FUNCTION accessible_recipients() RETURNS SETOF uuid ...
CREATE POLICY "..." USING (id IN (accessible_recipients()));

-- CORRECT — scalar boolean, called once per row
CREATE FUNCTION user_can_access_recipient(p_id uuid) RETURNS boolean ...
CREATE POLICY "..." USING (user_can_access_recipient(id));
```

**Helper functions** (all SECURITY DEFINER — run as postgres, not calling user):

| Function | What it checks |
|---|---|
| `user_can_access_recipient(uuid)` | Is the current user a member of this recipient's org with accepted_at set? |
| `user_is_coordinator_for(uuid)` | Is the current user a coordinator for this recipient's org? |
| `user_in_org(uuid)` | Is the current user an active (accepted) member of this org? |
| `user_is_org_coordinator(uuid)` | Is the current user a coordinator in this org? |

`SECURITY DEFINER` means these functions can join against tables the calling user
can't read directly (e.g., `memberships`), while still enforcing access control
inside the function body.

**Special cases:**

| Table | RLS behaviour |
|---|---|
| `identity_vault` | `USING (false)` for anon + authenticated — always returns 0 rows |
| `outer_circle_requests` | `USING (true)` for reads — intentionally open, token enforced in API |
| `care_briefs` | `USING (true)` for reads — intentionally open, token enforced in API |

## Supabase Client Decision Tree

```
Do you need to read/write identity_vault?
    YES → supabaseAdmin (server/ or app/api/ only)

Do you need to bypass RLS for any reason?
    YES → supabaseAdmin (server/ or app/api/ only)

Are you in a server component, server action, or API route?
    YES → createServerSupabase() for session-scoped reads
          supabaseAdmin for writes that need service role

Are you in a client component ('use client')?
    YES → createClient() (browser client, anon key)
```

## Invite Token Security

- Tokens are 32-byte random hex strings (64 hex chars), generated by `pgcrypto` in the DB
- Tokens expire after **48 hours** (`expires_at` column)
- Tokens are **single-use** (`consumed_at` is set on acceptance, checked before use)
- Tokens are **email-scoped** — the accepting user's email must match `invite_tokens.email`
- Email is stored and compared in lowercase to prevent case mismatch bypasses

## Outer Circle Token Security

`outer_circle_requests.share_token` is the only access control for the volunteer board.

- Anyone who knows the URL can view and claim slots — no account required
- The URL never exposes the care recipient's name, org name, or any identifying info
- Slot claiming is atomic via `claim_outer_circle_slot()` Postgres function
- Revocation: set `active = false` on the request — claims become inert

## Critical Index for RLS Performance

Every RLS policy helper function queries `memberships`. Without the right index,
every query involving an RLS check becomes a full table scan.

```sql
CREATE INDEX memberships_user_org_recipient_active
  ON memberships (user_id, org_id, recipient_id)
  WHERE accepted_at IS NOT NULL;
```

This index covers the most common RLS check pattern:
`WHERE user_id = auth.uid() AND org_id = ? AND accepted_at IS NOT NULL`
