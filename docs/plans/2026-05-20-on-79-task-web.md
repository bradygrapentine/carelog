## Goal

Make the ON-77 task foundation user-visible: a `tasks` tRPC router (create/list/update/complete/assign) gated by a TS port of the ON-77 permission model, plus a web Tasks panel (list + detail, free-text instructions, clickable checklist persisting to `checklist` jsonb, pin-to-shift, role-aware action gating). Unblocks ON-80 (mobile reuses the router), ON-82, ON-83.

## Critical design constraint (resolved at cycle-1 review — drives FIND-001/002/003)

**Use the RLS-scoped client `ctx.supabase` for ALL task CRUD — NOT `supabaseAdmin`.** This is the load-bearing decision. `ctx.supabase` carries the caller's JWT (shifts.ts:49: "ctx.supabase is RLS-scoped — filters to orgs the caller belongs to"), so `auth.uid()` is the real user. That makes the entire ON-77 authz surface — RLS policies AND the `tasks_update_guard` trigger (which calls `auth.uid()` + `user_can_complete_task`) — work exactly as designed and pgTAP-tested. Authz is therefore **delegated to ON-77's already-shipped RLS + trigger**; the router does NOT re-implement it.

Why this matters (the trap avoided): `supabaseAdmin` is service-role → `auth.uid()` returns NULL. If the router used admin, the trigger's `user_can_complete_task(OLD.id)` would evaluate against NULL and **raise `tasks_complete_forbidden` for every legitimate completion**, and `completed_by := auth.uid()` would stamp NULL. So the admin path is not just a parity risk — it's outright broken for tasks. There is **no TS port of the permission predicates** (no `taskPermissions.ts`); the single source of truth stays in SQL (ADR-0007). `supabaseAdmin` is used ONLY for the narrow assignee-validation read in `assign` (cross-membership lookup), if RLS can't serve it.

## Non-goals

- **No mobile** (ON-80), **no notifications** (ON-81), **no recurring/future-shift scheduling** (ON-82), **no recipient-initiated flow** (ON-83 — `requested_on_behalf_of` stays unused here).
- No new migration / schema change — consume the shipped ON-77 tables/types as-is. (If a `p_user_id` predicate variant is wanted later to unify SQL/TS authz, that's a separate row.)
- No changes to `task_permissions` editing UI (admin config surface is a later row) — the router reads it; v1 relies on the coordinator-only default.
- Do not touch unrelated routers, the ON-77 migration, or `BACKLOG.md`.

**Implementation steps:**
1. **`tasks.ts` router** (`protectedProcedure`, **`ctx.supabase`** — RLS-scoped; NOT `supabaseAdmin` except where noted). Authz is delegated to ON-77 RLS + trigger; the router validates input, sets caller-derived fields, and maps DB errors:
   - `list` (`.input` recipientId + optional shift_id/assignee filter) → `ctx.supabase.from("tasks").select(...)`. RLS already scopes to the caller's accessible recipients (FIND-002 — no manual org check needed; org-B sees zero rows).
   - `create` (`.input(createTaskPayload)`) → insert via `ctx.supabase` with `created_by`/`requested_by = ctx.user.id`, `status='todo'`. RLS `tasks_insertable_by_creator` enforces create-permission + recipient access; a denied insert surfaces as a Postgres RLS error → map to `FORBIDDEN`.
   - `update` (`.input(updateTaskPayload)`) → update via `ctx.supabase`; never send `completed_by`/`completed_at` (the trigger owns them). The `tasks_update_guard` trigger enforces content-edit authority; a RAISE → map to `FORBIDDEN`.
   - `complete` (taskId) → `ctx.supabase` update `status='done'`. Trigger gates completion (`user_can_complete_task`) and server-stamps `completed_by = auth.uid()` (real user under ctx.supabase). RAISE `tasks_complete_forbidden` → `FORBIDDEN`.
   - `cancel` (taskId) → `ctx.supabase` update `status='cancelled'`. Trigger gates coordinator-or-creator. RAISE → `FORBIDDEN`. (Explicit procedure so the UI doesn't overload `update`.)
   - `assign` (taskId, assigneeUserId|null) → first **read the task row server-side** (`ctx.supabase`) to get its `org_id`/`recipient_id` — never trust a client-supplied org (S1). Then `ctx.supabase` update `assigned_to` (trigger's content-edit gate governs WHO may assign). **Assignee validation:** the assignee must be able to access the task's recipient — validate against the task's server-read `org_id`/`recipient_id` via a single scoped `supabaseAdmin` `.from("memberships").select` (cross-user lookup), reject out-of-org/no-access assignee with `BAD_REQUEST`.
   - **Error/denial mapping (S2 — critical):** an RLS **USING** denial (e.g. updating a row the caller can't access) does NOT raise SQLSTATE `42501` — `ctx.supabase.update(...).select()` silently returns **0 rows / empty data with no error**. So the router MUST treat **0 rows affected on a mutation as `FORBIDDEN`**, not success. Separately, map trigger RAISE constants (`tasks_complete_forbidden`/`tasks_edit_forbidden`/`tasks_cancel_forbidden`/`tasks_immutable_column`) + INSERT WITH CHECK SQLSTATE `42501` → `TRPCError({code:"FORBIDDEN"})`. These are PHI-free constants; never pass a raw Postgres `error.message` into the TRPCError message.
2. **Router unit tests** (`tasks.logic.test.ts`): (a) every procedure uses `ctx.supabase`, NOT `supabaseAdmin` (the assignee-validation read is the only allowed admin call) — assert via the mock; (b) a simulated trigger RAISE / RLS `42501` from the client maps to tRPC `FORBIDDEN` for complete/update/cancel; (c) `complete`/`update` never include `completed_by`/`completed_at` in the payload sent to the client (FIND-003 — trigger owns them); (d) `create` sets `created_by`/`requested_by` = `ctx.user.id`; (e) `assign` rejects an out-of-org assignee. **Authz correctness itself (cross-org deny, role-gated completion) is already proven by the ON-77 pgTAP** (`supabase/tests/tasks.test.sql`) — these router tests prove client-selection + error-mapping, not re-prove RLS.
3. **UI `TasksPanel.tsx` + `TaskDetail.tsx`** per `.claude/rules/ui-standards.md`: tinted `CardHeader`, click-to-open create form (title + instructions textarea + checklist item add), task list, detail with clickable checklist toggles (persist via `update`), shift picker (current+upcoming shifts) for `shift_id`, role-aware gating (hide Complete/Cancel affordances the caller can't use — the server still enforces; UI-hiding is convenience only). `instructions` rendered as **plain text / sanitized** (no `dangerouslySetInnerHTML` — FIND-006). Keyboard + focus-ring + labels + tokens per ui-standards. **Checklist-toggle gating (S3):** the ON-77 trigger treats `checklist` as a content edit → only creator/coordinator/creator-role may toggle. v1 honors that (hide toggles for others; server enforces). If the product wants the *assignee* to check off items, that needs an ON-77 trigger carve-out — out of scope here; **seed a follow-up row** rather than widen scope.
4. **Component tests** (`TasksPanel.test.tsx`): renders list, create form opens/submits, checklist toggle calls mutation, Complete hidden for a role without completion capability.
5. Mount `<TasksPanel recipientId={...}>` in the journal recipient page (one import + one JSX line).

**Risk + mitigations:**
- *Router accidentally uses `supabaseAdmin` (breaks the trigger — completion fails / completed_by NULL)* → acceptance greps cap admin usage at ≤1 (assignee read); the cycle-1 review caught exactly this. RLS-scoped `ctx.supabase` is mandatory for CRUD.
- *Trigger RAISE leaks a raw PG string* → error mapper translates known constants → FORBIDDEN; no `error.message` passthrough.
- *Checklist toggle blocked for assignees* (S3) → v1 honors the ON-77 content-edit gate (creator/coordinator/creator-role); follow-up row seeded if assignee-checkoff is wanted (needs an ON-77 trigger carve-out — out of scope).
- *Stored XSS via instructions* (FIND-006) → plain-text/sanitized render.
- *PR too large* → server phase (steps 1–2) may split from UI (3–5) into two PRs if review strains; default one PR.
