## Discoveries during planning

**TD-155 closes as no-op.** Live-test "429 at attempt 2" came from Supabase Auth's built-in OTP send-rate limiter, NOT carelog's middleware:
- `SignInForm.tsx:46` calls `supabase.auth.signInWithOtp` directly client → Supabase Auth. No carelog code in the path.
- The carelog `rateLimit()` middleware (`apps/web/lib/rateLimit.ts`) DOES guard `/api/auth/verify` (lines 13-18), but that's OTP **verification**, not send.
- For verify, IP-keying is the correct choice — defends against rotating-IP brute force on a single victim's 6-digit code. Email-keying would weaken the gate by letting an attacker brute-force a known victim's code as fast as they can rotate residential proxies.
- Household-IP concern is bounded: max=30 per 15min on verify is plenty for any household, and SEND is Supabase's responsibility.

Deliverable for TD-155: one inline comment on `rateLimit.ts` explaining the IP-keying choice + close the row.

## Track A — TD-167 · Generic error responses on invite-accept paths

**Branch:** `fix/td-167-invite-error-message-generic`
**Files allowed:**

- `apps/web/server/routers/__tests__/membershipsRouter.security.test.ts` (existing file — add new test cases)

**Verified state at planning time** (HEAD `f84500d`):
- `route.ts:26-28` — RPC error branch returns `{ error: error.message }` with status 500. Raw Postgres error string echoed to client.
- `route.ts:43` — `messageMap[data.error] ?? data.error` fallback echoes raw sentinel string for any code not in the map (`not_found` / `email_mismatch` / `already_used` are in the map; future codes like `quota_exceeded` would leak).
- `route.ts:55-58` — catch branch returns `{ error: e.message }` with status 500. Same exposure.
- `memberships.ts:95-100` — tRPC mutation passes `wrapAdminError(error).message` as the TRPCError message.
- `memberships.ts:122-124` — fallthrough `else` branch passes `data.error ?? "Unknown error"` as the TRPCError message. Same class of leak.
- `trpc/index.ts:44` — default `errorFormatter({ shape }) { return shape }`. tRPC v11 default shape does NOT include `cause` in the client payload, so `cause:` is safe to use for rich diagnostics (server-only).

**Work:**
1. **HTTP route — RPC error branch (line 26-28).** Add `import * as Sentry from "@sentry/nextjs"` at the top of the file (currently no Sentry import). Replace branch with:
   - `Sentry.captureException(error, { tags: { component: "invite-accept", path: "rpc.error" } })`
   - `NextResponse.json({ error: "Internal error — please try again" }, { status: 500 })`
2. **HTTP route — sentinel fallback (line 43).** Change `messageMap[data.error] ?? data.error` to `messageMap[data.error] ?? "Request could not be completed"`. Documented sentinels (`not_found`, `email_mismatch`, `already_used`) still map to their existing user-facing strings; unknown future codes get the generic fallback. **AND** when the fallback fires, also call `Sentry.captureException(new Error(\`Unknown invite sentinel: ${data.error}\`), { tags: { component: "invite-accept", path: "sentinel.unknown" } })` so we surface the new code instead of silently swallowing it. Test case #4 asserts this capture.
3. **HTTP route — catch branch (line 55-58).** Same Sentry + generic-message pattern as step 1, with `path: "catch"` tag.
4. **tRPC mutation (memberships.ts:95-100).** Keep `wrapAdminError(error)` in the `cause:` slot (server-only — verified `trpc/index.ts:44` errorFormatter doesn't serialize `cause` to client). `message:` becomes `"Failed to accept invite"`. Add `Sentry.captureException(error, { tags: { component: "memberships.accept", path: "rpc.error" } })` before throwing.
5. **tRPC mutation — fallthrough branch (memberships.ts:122-124).** Same pattern: capture `data.error` as the Sentry message + tag `path: "rpc.fallthrough"`, throw with generic `message: "Failed to accept invite"`.
6. **PHI/security sanity:** Sentry captures pass the error object only — NOT `p_token`, `p_email`, or `p_user_id` in context. Note: `p_token` is a 64-char opaque token (not PHI per ADR-0001, but capturing it in Sentry would create a replay-attack surface so we deliberately exclude it from the breadcrumb data). User-id UUIDs are fine.
7. **Tests.** Four new vitest cases:
   - HTTP route, RPC error: when `supabaseAdmin.rpc` returns `{ error: { message: "anything" } }`, response body is exactly `{ error: "Internal error — please try again" }` AND `Sentry.captureException` was called once with the right tags.
   - HTTP route, catch path: when RPC throws, response body is the generic 500 AND Sentry captured.
   - tRPC mutation, RPC error branch: when RPC returns `error`, TRPCError's `message` is `"Failed to accept invite"` (NOT the raw string) AND Sentry captured with `path: "rpc.error"`.
   - tRPC mutation, fallthrough: when RPC returns `data.success=false` with an unknown error code, TRPCError's `message` is `"Failed to accept invite"` (NOT the raw `data.error`) AND Sentry captured with `path: "rpc.fallthrough"`.

**Acceptance:**

- `grep -nE '\bdata\.error\b|\berror\.message\b' apps/web/app/api/invite/\[token\]/accept/route.ts` returns 0 matches inside response-body slots (only Sentry-capture context is allowed).
- Same grep on `apps/web/server/routers/memberships.ts` (accept mutation block lines 88-127) returns 0 matches in the `message:` slot of any `TRPCError`.
- Full suite passes (~2160+ tests).
