## Goal

Make tasks usable in the web app: a Tasks destination in the journal sidebar with a list + detail panel — create tasks (title, instructions, checklist), toggle checklist items (gated to creator/coordinator/creator-role per the ON-77 trigger — NOT assignee in v1), pin to a shift, complete/cancel — all calling the shipped `trpc.tasks.*` router. Also fold in the 3 post-wave router test-gaps from #679. Completes the visible task loop; gives ON-80 (mobile) its UX-parity reference.

## Non-goals

- No mobile (ON-80), no notifications (ON-81), no recurring/future scheduling (ON-82), no recipient-initiated flow (ON-83).
- No `task_permissions` config-editing UI (v1 = coordinator-only default; later row).

## v1 gating limitation (explicit — must seed a follow-up)

UI affordances (create/complete/checklist-toggle) are gated to `currentUserRole === "coordinator"`. ON-77 lets an org **widen** `creator_roles`/`completer_roles` (e.g. to `caregiver`), so a *permitted* caregiver in a widened org would see NO button even though the server would allow the action — a real functional gap vs. the data model, not just cosmetic. **Acceptable for v1 ONLY because the `task_permissions` default is coordinator-only and no config-editing UI exists yet.** Seed a follow-up row (post-wave) to gate the UI off the org's actual `task_permissions` (a `trpc.tasks.capabilities` query or similar) once the config UI lands. The server remains the true gate regardless.
- No new migration / server router change beyond the `isForbidden` substring tidy + added tests.
- Do not touch `BACKLOG.md`, unrelated panels, or auth/proxy files.

## Threat/UI controls (fold into acceptance)

- **PHI (ADR-0001):** no task `title`/`instructions`/`checklist` content into `posthog`/`Sentry` — UUID-only. ESLint `carelog/no-phi-in-analytics` must stay clean.
- **XSS (FIND-006):** `instructions` rendered plain-text or sanitized — no `dangerouslySetInnerHTML` with raw user content.
