# Implementation Plan ‚Äî TD-144 atomic OCR confirm RPC

**Date:** 2026-05-17
**Slug:** `td-144-atomic-ocr-confirm`
**Source row:** TD-144 (P2 / Medium)
**Mode:** direct (1 track, ~3h)
**Threat model:** `.claude/state/owasp-threat-td-144.md` (0 Critical / 0 High; 3 Medium controls baked into acceptance below)

## Problem

`apps/web/app/api/ocr/confirm/route.ts` performs three sequential writes via separate `supabaseAdmin.from()` calls:

1. `INSERT INTO ocr_audit_log` (lines 100‚Äì113) ‚Äî append-only audit row per attempt.
2. `INSERT INTO medications` (lines 117‚Äì128) ‚Äî new medication row.
3. `UPDATE ocr_jobs SET status = 'confirmed' WHERE id = ... AND status = ...` (lines 131‚Äì147) ‚Äî optimistic-lock status flip.

Steps 1 + 2 commit immediately. Step 3 uses optimistic locking. If two requests with the same `jobId` race past step 1's audit insert and both reach step 2, **both insert medication rows**. The optimistic-lock at step 3 only protects `ocr_jobs.status` ‚Äî by the time the race-loser hits 409, its `medications` row is already committed.

SEC-007 (PR #492 era) deliberately keeps per-attempt audit semantics ‚Äî that's correct for forensics. The medication duplication is the real bug.

## Goal

Collapse audit + medication + status flip into one SQL transaction via a SECURITY DEFINER RPC. The route handler becomes a thin caller: Zod validation + coordinator-membership check + one `supabaseAdmin.rpc('confirm_ocr_job', ...)`. Sentinel error codes (mirrors `accept_invite` pattern) map to user-safe 4xx responses.

## SQL function shape

```sql
CREATE TYPE confirm_ocr_job_result AS (
  success boolean,
  error   text  -- 'not_found' | 'not_pending' | 'org_mismatch' | 'already_confirmed' | NULL
);

CREATE OR REPLACE FUNCTION confirm_ocr_job(
  p_user_id              uuid,
  p_org_id               uuid,
  p_job_id               uuid,
  p_drug_name            text,
  p_dosage               text,
  p_instructions         text,
  p_raw_output_hash      bytea,
  p_confirmed_field_keys text[]
) RETURNS confirm_ocr_job_result
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public, pg_temp
AS $$
DECLARE
  v_job    ocr_jobs%ROWTYPE;
  v_result confirm_ocr_job_result;
BEGIN
  -- T3: lock the job row before any writes. Concurrent caller waits here.
  SELECT * INTO v_job
  FROM   ocr_jobs
  WHERE  id = p_job_id
  FOR UPDATE;

  IF NOT FOUND THEN
    v_result.success := false;
    v_result.error   := 'not_found';
    RETURN v_result;
  END IF;

  -- T1: org must match the row, not just the caller's claim.
  IF v_job.org_id <> p_org_id THEN
    v_result.success := false;
    v_result.error   := 'org_mismatch';
    RETURN v_result;
  END IF;

  IF v_job.status = 'confirmed' THEN
    v_result.success := false;
    v_result.error   := 'already_confirmed';
    RETURN v_result;
  END IF;

  -- State machine: confirmed is reachable ONLY from needs_review.
  -- Source of truth: apps/web/lib/ocr/jobStateMachine.ts LEGAL_TRANSITIONS.
  IF v_job.status <> 'needs_review' THEN
    v_result.success := false;
    v_result.error   := 'not_pending';  -- sentinel kept for caller back-compat
    RETURN v_result;
  END IF;

  INSERT INTO ocr_audit_log (
    ocr_job_id, org_id_snapshot, user_id,
    raw_output_hash, confirmed_field_keys, field_count
  ) VALUES (
    p_job_id, p_org_id, p_user_id,
    p_raw_output_hash, p_confirmed_field_keys,
    COALESCE(array_length(p_confirmed_field_keys, 1), 0)  -- empty array ‚Üí 0, not NULL
  );

  INSERT INTO medications (
    org_id, recipient_id, drug_name, dosage, instructions, created_by
  ) VALUES (
    p_org_id, v_job.recipient_id, p_drug_name, p_dosage,
    NULLIF(p_instructions, ''), p_user_id
  );

  UPDATE ocr_jobs SET status = 'confirmed' WHERE id = p_job_id;

  v_result.success := true;
  v_result.error   := NULL;
  RETURN v_result;
END;
$$;

REVOKE EXECUTE ON FUNCTION confirm_ocr_job(uuid,uuid,uuid,text,text,text,bytea,text[]) FROM PUBLIC, anon, authenticated;
GRANT  EXECUTE ON FUNCTION confirm_ocr_job(uuid,uuid,uuid,text,text,text,bytea,text[]) TO service_role;

COMMENT ON FUNCTION confirm_ocr_job IS
  'TD-144: collapses audit insert + medication insert + status flip into one tx, eliminating the medication-duplication race in the prior 3-statement pattern at apps/web/app/api/ocr/confirm/route.ts.';
```

**State-validity set is LOCKED:** `confirmed` is reachable only from `needs_review`. Source: `apps/web/lib/ocr/jobStateMachine.ts:34` `LEGAL_TRANSITIONS` (`needs_review ‚Üí confirmed` is the only inbound edge to `confirmed`; all other states fail). Do NOT widen at implementation time without a new ADR.

**One read still happens before the RPC:** we need `raw_text` to compute the SHA-256. Options:
- (a) `SELECT raw_text FROM ocr_jobs WHERE id = $1 AND org_id = $2` (one read, then the RPC re-locks the row). Slight inefficiency, two row touches; race-safe because the RPC's `FOR UPDATE` is the only mutating path.
- (b) Pass `raw_text` plaintext into the RPC and hash inside Postgres via `digest(raw_text, 'sha256')` from `pgcrypto`. Cleaner but adds extension dependency.
- (c) Read raw_text inside the RPC, compute hash via `digest()`, eliminate the route-side `createHash` call. Same as (b) plus the extra read inside the tx ‚Äî most consistent.

**Pick (a)** ‚Äî keeps `pgcrypto` out, keeps the SHA-256 deterministic in TS (testable), and the pre-read is harmless because the RPC owns the atomic boundary. Document this decision in the migration's leading comment.

## Acceptance

1. **Migration applies cleanly:** `pnpm migration-check` reports no drift; `supabase db reset && supabase test db` passes locally.
2. **pgTAP contract test for the race-loser branch (NOT true concurrency).** `supabase test db` runs in a single connection ‚Äî cross-session concurrent transactions are infeasible (prior art: `supabase/tests/invite_tokens_rls.test.sql:196-200` documents exactly this limitation for `accept_invite`). Instead, prove the contract: call `confirm_ocr_job(...)` twice in sequence on the same job. **First call:** returns `success=true`, inserts 1 medication row + 1 audit row, flips status to `confirmed`. **Second call:** returns `success=false, error='already_confirmed'`, inserts 0 additional rows, status unchanged. This exercises the exact `v_job.status = 'confirmed'` sentinel that the `FOR UPDATE` lock would funnel a true concurrent loser through. ## Risks accepted

- **Pre-RPC read of `raw_text`** is a non-locking read; the RPC re-fetches the row under `FOR UPDATE`. If a malicious caller modifies `raw_text` between the read and the RPC, the audit hash will not reflect the row's current `raw_text`. Mitigation: `ocr_jobs.raw_text` is never updated post-insert in any current route (verify by grep before merge). If TD-X adds a `raw_text` UPDATE path later, this becomes a real concern.
- **`FOR UPDATE` on `ocr_jobs` can block another `FOR UPDATE` from `apps/web/app/api/ocr/discard/route.ts` or similar.** Verified non-issue: those routes don't use `FOR UPDATE`; they use the same optimistic-lock pattern. The new function is the only `FOR UPDATE` site on `ocr_jobs`, so no deadlock surface.
- **Migration name `20260517000000_confirm_ocr_job_rpc.sql`** uses today's date prefix ‚Äî verify nothing else has been merged with the same prefix before pushing (`ls supabase/migrations/2026051700*.sql`).

## Out of scope

- `ocr_jobs.status` doesn't get a CHECK constraint added ‚Äî keep the existing free-text-with-app-validation pattern. Tightening the column to an enum is its own backlog row.
- `medications.created_by` doesn't get a FK constraint added if missing ‚Äî separate TD.
- Per-row tenant audit on `medications` table ‚Äî existing RLS suffices; no new audit surface in this change.
