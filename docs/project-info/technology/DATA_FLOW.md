# Carelog — Data Flows

## 1. Care Event Write Path

Every user action that produces a record follows this path.

```
JournalEntryForm (browser)
        │
        │  handleSubmit():
        │    1. Read form values BEFORE any await  ← React synthetic event rule
        │    2. POST /api/journal { recipientId, orgId, text, mood, userId }
        │
        ▼
POST /api/journal/route.ts  (Next.js API route)
        │
        │  Uses supabaseAdmin (service role) directly
        │  No Zod validation here — raw insert  ← Tech Debt #3
        │
        ▼
care_events table
  org_id, recipient_id, actor_id
  event_type: 'journal'
  entry_kind: 'human'
  payload:    { text, mood }    ← jsonb
  occurred_at: now()
  flagged:    false
        │
        │  JournalClient refreshes via GET /api/journal?recipientId=...
        │  (full reload, no optimistic update)
        ▼
JournalTimeline re-renders with new event at top
```

**The validated path (repositories):** `careEventsRepository.insertEvent()` runs
`validatePayload(eventType, payload)` before any DB write. The journal API route
bypasses this and inserts directly — a known tech debt. The repository-based path
should be preferred for new event types.

```
validatePayload(type, payload)    ← packages/schemas/src/careEvents.ts
        │
        │  Looks up payloadSchemas[type]
        │  Calls schema.parse(payload)
        │  ZodError thrown → never reaches DB
        │
        ▼
supabase.from('care_events').insert(...)
```

## 2. Identity Resolution Path (Display Name Cache)

Real names are never stored in `care_recipients`. To display a name:

```
Component needs recipient's name
        │
        ▼
resolveAndCacheDisplayName(recipientId, orgId, token)
        │
        │  1. Query display_names WHERE recipient_id = ? AND expires_at > now()
        │
        ├── Cache HIT (within 24h) ────────────────────────────────────►  return full_name
        │
        └── Cache MISS ─────────────────────────────────────────────────►
                │
                │  2. Query identity_vault WHERE token = ? AND org_id = ?
                │     Requires supabaseAdmin (service role)
                │     anon/authenticated roles get 0 rows
                │
                ▼
         identity_vault.full_name  (PHI)
                │
                │  3. Upsert display_names:
                │     { recipient_id, org_id, full_name,
                │       expires_at: now() + 24h }
                │
                ▼
         return full_name  (cached for next 24h)
```

**Why the cache exists:** A timeline with 50 events previously required 50 vault
reads (one per recipient lookup). The `display_names` table reduces this to one
cache read per recipient per 24 hours.

**Who can write the cache:** Only `supabaseAdmin` (service role). RLS on
`display_names` allows authenticated users to read only records for recipients
they can access, but writes are service-role-only.

## 3. Invite Write Path

```
TeamPanel (browser)
        │
        │  handleInvite():
        │    POST /api/invite { orgId, recipientId, role, email, invitedBy }
        │
        ▼
POST /api/invite/route.ts
        │
        │  1. Validate role is in allowed list
        │
        │  2. Insert memberships row:
        │       user_id = NULL       ← pending state (no user yet)
        │       accepted_at = null
        │
        │  3. Insert invite_tokens row:
        │       membership_id → links back to membership
        │       email = lowercase email
        │       token = generated by DB trigger (32-byte hex)
        │       expires_at = now() + 48h
        │
        │  4. Return inviteUrl = 'http://localhost:3000/invite/' + token
        │     TODO: send via Resend in production
        │
        ▼
Coordinator copies URL and sends manually (local dev only)
```

## 4. Invite Accept Path

```
POST /api/invite/[token]/accept
        │
        │  Inputs: { userId, userEmail }
        │
        │  Validations (fail fast):
        │    1. Token exists in invite_tokens
        │    2. consumed_at IS NULL  (not already used)
        │    3. expires_at > now()   (not expired)
        │    4. invite.email === userEmail (case-insensitive match)
        │
        │  If all pass:
        │
        │  Promise.all([                   ← both updates together
        │    invite_tokens.update({ consumed_at: now() })
        │    memberships.update({ user_id: userId, accepted_at: now() })
        │  ])
        │
        │  Wrapped in Postgres function `atomic_invite_accept()` via
        │  migration 20260407000000_atomic_invite_accept.sql — true transaction.
        │
        ▼
Membership is now active. User appears in team panel.
```

## 5. Onboarding Write Path

First-time setup creates 4 rows across 4 tables in sequence:

```
POST /api/onboarding/create
        │
        │  Input: { recipientName, recipientDob, orgName, userId }
        │
        │  Sequential (each step depends on the previous):
        │
        │  1. createOrganization({ name, orgType: 'family' })
        │     → organizations.id
        │
        │  2. createIdentity({ orgId, fullName, dob })
        │     → identity_vault.token
        │
        │  3. INSERT care_recipients { org_id, identity_token }
        │     → care_recipients.id
        │
        │  4. INSERT memberships { org_id, user_id, role: 'coordinator',
        │                          accepted_at: now() }
        │
        │  5. UPDATE user_profiles SET onboarded = true
        │
        ▼
User can now access /dashboard and /journal/[recipientId]
```

## 6. Care Event Read Path (Timeline)

```
JournalClient mounts
        │
        │  GET /api/journal?recipientId=...
        │
        ▼
GET /api/journal/route.ts
        │
        │  supabaseAdmin.from('care_events')
        │    .eq('recipient_id', recipientId)
        │    .order('occurred_at', { ascending: false })
        │    .limit(50)
        │
        ▼
JournalTimeline receives events[]
        │
        │  For each event:
        │    entry_kind === 'human'  → full card with mood badge + text
        │    entry_kind === 'system' → compact row
        │
        │  Time formatting:
        │    same day  → "2:34 PM"
        │    yesterday → "Yesterday 2:34 PM"
        │    older     → "Mar 15, 2:34 PM"
        │
        ▼
Rendered timeline (most recent at top)
```
