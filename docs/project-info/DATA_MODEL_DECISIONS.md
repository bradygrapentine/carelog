# Carelog — Data Model Design Decisions

This document explains WHY the data model is the way it is.
Before changing any schema, read the reasoning here first.

## The identity vault pattern

**What:** Real names, DOB, and contact info are stored in `identity_vault` only.
All other tables use a UUID token to reference this data.

**Why:** HIPAA-adjacent data handling. The identity vault is accessible by
service_role only — anon and authenticated Supabase roles return zero rows.
This keeps PHI out of logs, error reports (Sentry), analytics (PostHog),
and background jobs (Inngest) — none of which have a Supabase service role key.

**What this means in practice:**
- `care_recipients.identity_token` → UUID that maps to `identity_vault.token`
- Resolving a name requires a server-side call with the service role key
- The `display_names` table is a 24-hour cache to reduce repeated vault reads
- The `care_briefs` table de-tokenizes once at creation time

**What you cannot do:**
- SELECT real names from care_recipients (they're not there)
- Display names in a client component without a server-side resolution step
- Log care recipient names to any external service

## The care_events universal log

**What:** Every action taken in the platform creates a care_event row.
Journal entries, medication logs, shift completions, expense records —
all go into the same table.

**Why:** A single longitudinal record enables the full history export,
the weekly digest, the doctor export, and the burnout analysis.
If these were separate tables, generating a complete timeline would
require joining many tables with complex ordering.

**The entry_kind column:**
- `'human'` — written by a person. Journal entries, handoff notes.
  Displayed prominently in the timeline.
- `'system'` — generated by the platform. Medication given, shift completed.
  Displayed compactly in the timeline.

This distinction is UX-critical. Human entries drown in system noise
without it. The original design didn't have this — it was added after
competitor analysis showed CaringBridge's problem with system-generated
noise overwhelming human content.

**The payload column:**
jsonb, validated by Zod schema per event_type before insert.
Each event_type has its own schema in `packages/schemas/src/careEvents.ts`.
The `validatePayload(eventType, payload)` function is called before
every insert. An invalid payload throws a ZodError — it never reaches the DB.

**The generated missed column:**
```sql
missed boolean GENERATED ALWAYS AS ((payload->>'missed')::boolean) STORED
```
This allows fast queries like "all missed medications this week" without
jsonb operators. Computed at write time, indexed.

## Why medications are NOT in care_events payload

The original design stored medication data as jsonb inside care_events.
This was changed after realizing:

1. "How many days of Lisinopril remain?" requires parsing jsonb, not querying a column
2. Refill alerts need `WHERE supply_days_remaining <= 7` — impossible with jsonb
3. The OCR pipeline needs a structured target to populate — jsonb blob is wrong shape
4. Gap detection on medication schedules needs clean data to query

The medications table is now the source of truth for the medication catalog.
care_events still records THAT a medication was given/missed, but references
the medication by ID not by name/dosage. The structured data lives in medications.

## Why recipient_id on memberships is nullable

NULL recipient_id = org-wide role (coordinator, supervisor)
Value recipient_id = recipient-scoped role (aide assigned to specific client)

This enables the agency model without any schema migration:
- A home care agency is one organization
- Each client family is one care recipient within that org
- Aides are scoped to specific recipients
- The agency coordinator sees all recipients

If recipient_id were always required, agencies would need a separate org per client,
which would break cross-client coordination and reporting.

## The outer circle design

outer_circle_requests deliberately has NO effective RLS protection.
The table has RLS enabled, but the policy is `USING (true)` — open read.

This is intentional. The outer circle is a volunteer board. Anyone with
the share_token can see the request. Anyone can claim a slot — no account required.

The share_token IS the access control. The URL `/help/{share_token}` never
exposes the care recipient's name, org name, or any identifying information.

The `claim_outer_circle_slot()` function is atomic:
```sql
UPDATE outer_circle_requests
SET slots_filled = slots_filled + 1
WHERE id = p_request_id
  AND active = true
  AND slots_filled < slots_total;

IF NOT FOUND THEN RAISE EXCEPTION 'slot_unavailable'; END IF;
```
One UPDATE wins. The second caller finds the WHERE clause fails and gets
the exception. No double-booking is possible.

## The care_briefs snapshot model

care_briefs stores a de-tokenized JSON snapshot, not a live query.

**Why not a live view?**
A live view would require the vault to be accessible at view time.
The care brief URL is shared outside the platform — with a doctor's receptionist,
a new facility, a discharge planner. These people don't have platform accounts.
Giving them a URL that triggers vault access would require service-role
credentials to be accessible from a public endpoint. That's a security disaster.

**The snapshot model:**
1. Coordinator requests a care brief
2. Server resolves the vault ONCE (service role, server-side)
3. The resolved data is stored as jsonb in care_briefs.content
4. share_token is generated, URL is returned
5. Anyone with the URL can view content — no auth, no vault access
6. Revoke = set revoked=true — the URL immediately returns 404

**Staleness:** The snapshot is point-in-time. If medications change after the
brief is generated, the brief is outdated. This is intentional — it's a document,
not a live feed. Coordinators regenerate it when handing off to a new provider.

## Why RLS uses scalar boolean functions

Postgres RLS policy expressions cannot call set-returning functions.
A function that returns `SETOF uuid` cannot be used in `USING (id IN (fn()))`.
This fails with SQLSTATE 0A000.

The solution is scalar boolean functions:
```sql
-- scalar — works in RLS
CREATE FUNCTION user_can_access_recipient(p_id uuid) RETURNS boolean ...
CREATE POLICY "..." USING (user_can_access_recipient(id));
```

These functions are `SECURITY DEFINER` — they run as the function owner (postgres)
not the calling user. This allows them to join against tables the calling user
can't directly read. The security boundary is enforced inside the function.

## Index strategy

Every query pattern that runs in a hot path has an index.

**care_events indexes:**
- `(recipient_id, occurred_at DESC)` — timeline queries
- `(org_id, event_type)` — filtering by type within an org
- `(actor_id, occurred_at DESC)` — activity by specific user
- `(recipient_id, event_type, occurred_at DESC)` — filtered timeline
- `(recipient_id, missed) WHERE missed = true` — partial index for missed meds

**memberships index:**
- `(user_id, org_id, recipient_id) WHERE accepted_at IS NOT NULL`
  This is the most critical index in the database. Every RLS policy function
  calls a query that joins against memberships. Without this index, every
  RLS check is a table scan.

**Partial indexes throughout:**
Many indexes use `WHERE active = true` or `WHERE accepted_at IS NOT NULL`.
This keeps the index small and fast — the vast majority of queries only
care about active/accepted records.

## The display_names cache

Problem: timeline rendering at 50 events requires 50 vault reads to resolve names.

Solution: `display_names` table with 24-hour TTL.
- Service role writes (upsert on cache miss)
- Authenticated role reads (RLS-scoped to accessible recipients)
- The vault is still the source of truth — the cache is a read-through layer

When to invalidate: when `identity_vault` changes. Currently manual.
Production fix: trigger on identity_vault UPDATE that deletes the cache entry.
